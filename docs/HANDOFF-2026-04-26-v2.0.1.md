## Was wurde geliefert

**v2.0.1 (PR #21, Squash `ca8ff24`):** 7 spec'd Scope-Items komplett.

**S1 — Storage-Forecast-UI (AC9-Closure):**
- Tauri-Commands `estimateStreamerFootprint` + `estimateGlobalFootprint`
- Streamer-Add-Dialog Forecast-Box mit Watermark-Risk-Indicator
- Settings-Page "Storage Outlook"-Section mit globalem Forecast + Per-Streamer-Breakdown

**S2 — Library-UI-Re-Konzeption (AC10-Closure):**
- Unified List "available + downloaded" mit Status-Badges
- Visuelle Differenzierung (Opacity, Border, Color je Status)
- Filter-Chips: All / Available / Downloaded / Watched
- Per-VOD Quick-Actions State-aware (Download / Watch / Remove)
- Event-getriebene UI-Updates via distribution:* Events
- Neuer Command `removeVod` mit Library-Root-Prefix-Guard (gefunden durch R-RC-03 Cross-Awareness)

**S3 — Pre-Fetch-Player-Wiring (AC8-Closure):**
- Player-Hook bei Watch-Progress > 70% oder remaining < 120s
- Throttle: max 1 Pre-Fetch pro VOD pro Session (app-session via module-level Set)
- Tauri-Command `prefetchCheck`

**S4 — Windows-CPU-Suspend (AC5-Closure):**
- Windows-Pfad via PowerShell + `Add-Type` P/Invoke (statt windows-rs Crate)
- Begründung: behält `unsafe_code = "forbid"` bei, kostet ~0.3-0.8s Startup-Latenz pro Call
- Cross-Platform Trait `ProcessSuspendController` mit Unix (SIGSTOP) + Windows-Impls

**S5 — Download-Worker State-Convergence:**
- Worker beobachtet `vods.status = 'queued'` via Event-Sink-Bridge in lib.rs
- 5s-Tick-Latency on first download akzeptiert (Service-Decoupling-Vorteil überwiegt)
- queued → downloading State-Transition wird vom Worker getriggert

**S6 — WatchEvent::Completed Wiring:**
- Watch-Service Completed-Event → distribution.on_watched_completed
- Auto State-Transition ready → archived
- Sliding-Window-Enforce-Trigger danach

**S7 — Stale-PID-Guard für SuspendController:**
- Pre-Suspend/Resume Process-Liveness-Check (kill -0 Unix, GetExitCodeProcess Windows)
- Silent no-op bei toten PIDs statt Crash
