## 2026-04-26 — Sub-Phase A: Windows suspend via PowerShell (not `windows-rs`)

The mission text suggested `windows-rs` for `SuspendThread` /
`ResumeThread`, but the workspace lints set `unsafe_code = "forbid"`
which would block an FFI-direct implementation without flipping the
lint workspace-wide.  Instead, `WindowsSuspend` shells out to
`powershell.exe` with an `Add-Type` P/Invoke shim that calls
`NtSuspendProcess` / `NtResumeProcess` via `ntdll.dll`.  This matches
the existing `infra::process::priority` pattern (`wmic CALL
SetPriority`) — same shell-out idiom, same lint compatibility.

Trade-off: each call pays ~0.3–0.8 s of PowerShell startup +
JIT-compile cost.  The throttle samples at 5 s intervals so this
is invisible to the user.  A native `windows-rs` rewrite is filed
as a v2.1 follow-up if support data shows the latency matters.

R-RC-01 + R-RC-02 both CLEAN after one re-review iteration (P1
fixes for locale-independent ESRCH detection + try/finally on the
PowerShell close path).

## 2026-04-26 — Sub-Phase B: prefetch throttle scope = app-session, not component-mount

R-RC-01 caught a doc-vs-implementation drift: the original hook
used `useRef` to hold the "already triggered" set, which would reset
on every component remount (player open / close / re-open of the
same VOD would re-fire pre-fetch).  ADR-0031's intent is "per
session" — meaning the whole app session, not just the player
instance.  Promoted the set to a module-level singleton.  Survives
HMR (Vite keeps module instances), resets on full app reload.

Also added `MIN_WATCHED_SECONDS_BEFORE_PREFETCH = 5` to gate both
trigger arms behind a "user has actually watched something" check,
because a 90 s clip mounted at `currentSeconds = 0` would otherwise
satisfy the remaining-floor branch immediately.

## 2026-04-26 — Sub-Phase C: vods.status convergence via direct UPDATE, not events

Two architectural options for the convergence (S5):

**Option A.** Have `pick_vod` in distribution call `enqueue` on
downloads, OR have downloads directly call distribution methods
when its state transitions.

**Option B.** Bridge in `lib.rs` via the existing `distribution_sink`
event handler — when `VodPicked` fires, call `downloads_svc.enqueue`
in a spawned task.  When `downloads.state` transitions, update
`vods.status` via direct SQL inside the `downloads` service.

Chose Option B.  Keeps services decoupled (no service-to-service
import of the other service's struct), matches the existing event-
driven integration pattern in `lib.rs`, and minimises blast radius.
The trade-off is: between `pick_vod` flipping `vods.status='queued'`
and the downloads worker observing the row, there's a 5 s window
where the worker hasn't created the corresponding `downloads` row
yet.  Acceptable for v2.0.1; documented in the commit message.

R-RC-02 re-review caught 3 bypass paths (cancel, retry,
crash_recover) that didn't go through `set_state`.  Refactored to a
unified `target_vods_status_for_state(state, cancel)` helper that
every transition path now calls.  User-cancel distinguished from
auto-failure: cancel → `available` (the user said no), auto-failure
→ `queued` (still wanted, retry-friendly).  `crash_recover`
wrapped in a transaction for atomic dual-table convergence.

## 2026-04-26 — Sub-Phase D: ForecastService takes Arc<dyn Clock>

R-RC-01 flagged a `commands/forecast.rs::unix_now()` helper that
read `SystemTime` directly, violating the `rust-backend.md` rule
("Clock-dependent code injects a `Clock` trait; never read
wall-clock time directly from domain/services code").  Refactored
`ForecastService::new` to accept `Arc<dyn Clock>` and read time
internally.  The command no longer threads `now_unix` as a
parameter — clean, test-deterministic via `FixedClock`.

The pre-existing `commands/app.rs::unix_now` predates the rule and
is filed as a separate follow-up cleanup.

## 2026-04-26 — Sub-Phase E: removeVod IPC instead of routing through cancelDownload

ADR-0033's "Remove" quick action for ready VODs was originally
specced as a new `removeVod` command.  R-RC-01 correctly flagged
that the initial commit declared the `onRemove` prop on `VodCard`
but never wired it from `LibraryPage` and never built the backend
command — making the Remove action unreachable.

Two options to close:
**Option A.** Skip the Remove button (scope-reduce AC10).
**Option B.** Build the missing `cmd_remove_vod` end-to-end.

Per R-SC-01 (no scope reduction without CTO approval), went with B.
`services::distribution::remove_vod` validates ready/archived →
deleted, unlinks the file via `tokio::fs::remove_file` (best-
effort, NotFound is benign), updates the downloads row to
`failed_permanent` only when state was `completed` (defensive
narrowing), and emits `VodArchived`.  4 service tests added.

A semantic follow-up was raised by R-RC-02: `remove_vod` reuses
`DistributionEvent::VodArchived` for the user-removal case — works
today (the renderer subscribes to the same topic for cache
invalidation) but conflates two distinct lifecycle events.  Filed
as a v2.1 cleanup to add `VodDeleted` or rename the enum variants.

## 2026-04-26 — Lifecycle filter chips operate client-side

ADR-0033 doesn't specify whether the lifecycle filters are
server-side (a `vods.status IN (...)` SQL filter on the listVods
query) or client-side (filter the already-fetched list).  Chose
client-side because:

1. The library page already loads the full per-streamer list with a
   200-row limit; filtering 200 cards in JS is microsecond-scale.
2. The status changes frequently (events fire, downloads complete);
   server-side filtering would cause more round-trips on every
   transition.
3. The test surface stays simple: assert the rendered list, not
   the SQL.

The legacy ingest-status filter (`eligible | skipped_*`) was
removed because the new lifecycle chips are the canonical filter
axis per ADR-0033.  A user who wants to filter by ingest status
can still do so via the API; the UI surface narrowed deliberately.
