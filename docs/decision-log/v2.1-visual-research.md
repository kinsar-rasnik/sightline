## 2026-05-11 — Tools for `visual-research`: Read/Write/Grep/Glob only

The brief specified `tools: [Read, Write, Grep, Glob]` — no Bash, no Edit. Held that strictly even though Edit would have been convenient for iterative anchor refinement. Reason: the brief's intent is unmistakable — this agent is research-only, never product-touching. Granting Edit would create the surface for a future operator to invoke `visual-research` and have it touch component code "while it's there." A read-and-write-greenfield-docs tool surface enforces the boundary at the tool layer, not just the prompt layer.

Trade-off: future refinements to an anchor document (e.g. "update H-07 to allow accent in the live-now badge") will require either re-invoking with a fresh Write of the full file, or the Senior Engineer doing the edit by hand. Accepted — this is a low-frequency operation; the boundary clarity is worth more than the convenience.

## 2026-05-11 — Anchor in English, CHANGELOG and decision-log in German

The repo's pattern: agent definitions, ADRs, and `docs/reference/*` are in English; `.claude/CHANGELOG.md`, decision-log entries, and `STATE.md` are in German. Followed both. The Netflix anchor document is in English because it is going to feed into v2.1 ADRs (`Timeline-Layout`, `VOD-Card`, `Visual-System v2`) which are also in English, and a future `@vision-compliance-reviewer` subagent that grades against the heuristics will want one consistent surface. The CHANGELOG / decision-log / STATE entries stayed in German to match every prior entry in those files.

No CEO call needed — pattern was clear from existing repo voice.

## 2026-05-11 — Heuristic count target: 25 (top end of 18–25 range)

The brief specified "18–25 prüfbare Regeln." Picked 25 because the Netflix language has enough distinct axes (layout, colour, typography, cards, hover, motion, chrome restraint) that compressing to 18 would have forced two-or-more concerns into single heuristics — which then become hard to grade pass/fail cleanly. With 25, each rule asserts one thing and tests one thing.

Distribution across categories: Layout 4, Color 4, Typography 4, Cards 4, Hover 4, Motion 2, Chrome 3 — roughly balanced, no single category dominates. A future reviewer that wants to weight categories can do so without re-numbering.

## 2026-05-11 — No screenshots; textual pattern descriptions only

The brief explicitly forbid embedding Netflix screenshots (copyright). Held strictly. Two consequences:

1. The component-pattern section (§3) is prose-only — aspect ratios named, idle/hover/focus states described, metadata composition enumerated. Took the position that "Netflix-faithful prose" is more durable as a spec anyway: screenshots date the moment Netflix re-skins, prose describes the stable underlying structure.
2. No reference URLs included even though `netflix.com` is the canonical surface. Reason: an anchor document that says "go look at netflix.com" leaves the reviewer to interpret what they see; an anchor document that says "Hero occupies ≥ 60 % of viewport height at scroll-0, with a bottom-to-mid gradient" is gradable without a live external reference. If the CEO later wants a reference-URL annotation, that can be added — but the default is self-contained.

## 2026-05-11 — 10 Open Questions for the CEO, not 5

The brief's example open-questions section sketched a few; the actual draft surfaced 10 places where Netflix language meaningfully collides with Sightline's domain (sidebar posture, hero-source, hover-preview-source-without-trailers, rail-axis, multi-perspective indicator, keyboard hover behaviour, retention-countdown threshold, multiview leader prominence, storage-outlook posture, continue-watching-rail existence).

Resisted the temptation to consolidate or pre-resolve any of them. Anker-First-Disziplin per the brief means: where the anchor and the domain genuinely diverge, the CEO decides, not the agent. Pre-resolving Q-3 (hover-preview source) by picking "first 30s of the VOD" would have been ergonomic but constitutes a v2.1 spec decision the brief explicitly defers.

---

## 2026-05-11 — Settings page deliberately out of anchor's browse-pattern scope

`SettingsPage` is in the Sightline-mapping table but only for chrome-level heuristics (BG, surface, accent, typography, focus, icons). Heroes, cards-on-hover, rails, detail-overlays, and motion rules are explicitly dropped — settings is utility, not browse. Called this out in the mapping table and noted that form chrome (inputs, toggles, selects) is governed by a separate convention not covered here.

Reason for the boundary: Netflix's settings panel is a thin form on a dark surface. There is no rail-language, no hover-preview-language, no hero-language. Pretending those apply would force the reviewer to grade `SettingsPage` as failing rules that were never meant for it. Cleaner: tell the reviewer "stop here."
